## Review-driven hardening of the movie_details/credits spans (Phase 1c-6/1c-7 follow-up)
Files: api/main.py
Why: Self-review of the initial 1c-6/1c-7 instrumentation surfaced five issues; this addresses the five the user accepted.

### Changes
- **#1 Unexpected-error visibility.** The payload_creation span disables auto status/record (so expected 404s stay UNSET), but that also suppressed marking on UNEXPECTED errors (DB error, malformed payload) — the span containing the failing op read green. Now wrapped in `try/except HTTPException: raise / except Exception: set ERROR + record_exception + raise`, so genuine failures always mark the span.
- **#2 No magic strings.** Span-attribute keys are now `_ATTR_*` module constants and the 404 reason values are a `NotFoundReason(str, Enum)` — the two endpoints share one definition, so a typo can't silently split a metric (satisfies the no-magic-strings coding standard). - **#3 Unified attribute key.** `movie_details.source` / `movie_credits.source` → one shared `movie.payload.source` (and `movie.not_found_reason`). Entity/artifact/dimension namespacing reads correctly on any span, the key names the dimension so new sources (postgres, another upstream) only add enum values, and cross-endpoint cache-hit-rate is now a single-key query (per-endpoint split via http.route / span name). Values stay role-based (`cache` not `redis`).

## Centralized telemetry naming system + full migration
Files: observability/names.py (new), api/main.py, observability_context/observability_architecture.md
Why: manual span names and attribute keys were raw string literals scattered across api/main.py (six `_ATTR_*` constants + inline span-name/`title_search.*` literals). The root token (`movie_credits`) was retyped in the span name and every attribute key — exactly what let the earlier `credits.*` vs `movie_credits.*` drift slip in. We wanted each namespace root written once, every name derived from it, and a documented ruleset so names can grow in any direction without migration.
Approach: new dependency-free `observability/names.py` (stdlib only — no otel import, so search/domain modules could reference a name without pulling in the SDK) with a `Name(str)` subclass. A Name *is* its dotted string (hands straight to `start_as_current_span`/`set_attribute`, no unwrapping) and `.child(segment)` derives a deeper name — so any token, root or leaf, can grow subtokens later without ever being a raw string. Every manual name is now a derived constant (`MOVIE_TMDB_ID = MOVIE.child("tmdb_id")`, etc.), imported by api/main.py. Chose `Name(str)` over (a) flat composed constants — still greppable but no shared-root structure and verbose — and (b) an enum — enums model closed VALUE sets (kept for `MoviePayloadSource`/`NotFoundReason`), not identifier namespaces; `.value` noise for no safety gain.
Behavior-visible change: exactly one emitted name changed — `movie.payload.source` → `movie.payload_source` (flattened per the "a namespace level must group ≥2 emitted attributes or have nameable roadmap siblings" rule; `payload` groups only `source`). Nothing in prod, so no data migration. All span names and every other attribute key are byte-identical.
Scope-out (deliberate): span EVENT strings (`"cache read failed"`, …) stay inline — human-readable messages, not queryable keys. Enum VALUES stay enums. No tests touched (the only repo match was a Python `.tmdb_id` attribute access, not a name-string assertion).
Design context: the naming ruleset (A structure / B root / C dot-vs-underscore / D leaf / E values-not-names / F cardinality) is codified in names.py's module docstring (source of truth) and summarized in observability_architecture.md §5.1; §5.2 keeps the instrumentation-behavior conventions; §6/§7/§9 updated (catalog now `movie.payload_source`, building-blocks row now the `Name` registry, conventions-promotion note points at names.py).
Testing notes: py_compile + full `import api.main` pass; grep guards confirm no `_ATTR_`, no inline name literals, and no 3-level `movie.payload.source` remain in code. Live Tempo re-verification (span names unchanged, attribute now reads `movie.payload_source`) still pending — testing paused by the user.

## Per-request outcome attributes (`outcome.success` / `outcome.failure_reason`)
Files: api/outcome.py (new), api/main.py, observability/names.py, observability_context/observability_architecture.md
Why: every instrumented endpoint should report a uniform, queryable verdict on the request. Previously only the two 404 sites recorded anything (`movie.not_found_reason`); the 502 and unexpected-500 paths set nothing on the server span, so "did this request succeed, and if not why" wasn't answerable in one place. "Failure" is broader than "error" — an un-indexed id is a failure but not a span error.
Approach: added `outcome.success` (bool, every path) + `outcome.failure_reason` (set only when false) to `observability/names.py` under a new `OUTCOME` root (dot-namespaced per rule C: it groups ≥2 emitted attributes). New `api/outcome.py` holds the mechanism: `FailureReason` str-Enum (coarse, actionable classes: `invalid_parameters`/`not_indexed`/`tmdb_removed`/`tmdb_fetch_failed`/`internal_error`), `EndpointFailure(HTTPException)` carrying a reason, and the `@record_outcome` decorator that writes the verdict on the server span ONCE. Chose top-level record-once + bubble-up (each failure site raises `EndpointFailure(failure_reason=…)`, the decorator records it) over per-site writes — same single-codepath-for-writes rationale as the naming system; a known failure declares *what* failed, the decorator owns *how* it's recorded. Kept `FailureReason` deliberately coarse (merged the two title_search validation reasons into `invalid_parameters`) since the span's other attributes carry the specifics.
Key simplification: `_fetch_movie_payload` no longer takes `request_span` — its only use was setting `not_found_reason`, which is now carried on the raised `EndpointFailure`. Its `except HTTPException: raise` still propagates `EndpointFailure` cleanly (subclass), so 404s stay UNSET and the 502 still marks the child span ERROR before raising.
Layering: names in `observability/names.py` (dependency-free) ← mechanism in `api/outcome.py` (imports FastAPI + OTel, so it lives under `api/`, not `observability/`) ← handlers in `api/main.py` (`@record_outcome` + `raise EndpointFailure`). Replaced `NotFoundReason` (deleted from main.py) with `FailureReason`.
Behavior-visible telemetry change: `movie.not_found_reason` is gone, replaced by `outcome.success`=false + `outcome.failure_reason`=`not_indexed`/`tmdb_removed`. Nothing in prod → no migration.
Scope-out: framework-level 422s (missing/mis-typed params rejected by FastAPI before the handler) carry no `outcome.*` — would need an app-level exception handler; deferred and noted in architecture doc §8. Enum VALUES stay enums; no tests touched.
Design context: outcome contract + bubble-up mechanism documented in observability_architecture.md §5.2; catalog (§6) now lists `outcome.*` as universal; building blocks (§7) list the `api/outcome.py` trio; names.py docstring rules C/E/F updated.
Testing notes: py_compile + `import api.main` pass; grep guards confirm no `MOVIE_NOT_FOUND_REASON`/`NotFoundReason` remain. Live Tempo verification pending (testing paused): `/movie_details/<bad-id>` should show `outcome.success=false`, `outcome.failure_reason=not_indexed`; a 200 should show `outcome.success=true` and no failure_reason.
