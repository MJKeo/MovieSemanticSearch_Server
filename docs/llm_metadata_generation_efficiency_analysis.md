## 1. Input Duplication Across Calls <a name="input-duplication"></a>

The same base movie data gets serialized and sent to multiple LLM calls independently. The table below shows which inputs go to which calls:

| Input | plot_events | plot_analysis | viewer_exp | watch_ctx | narr_tech | prod_kw | src_insp | reception |
|-------|:-----------:|:------------:|:----------:|:---------:|:---------:|:-------:|:--------:|:---------:|
| `title` | x | x | x | x | x | x | x | x |
| `overview` | x | x | | x | | | | |
| `plot_synopsis` (Wave 1 output) | | x | x | | x | | x | |
| `plot_summaries` (raw IMDB) | x | | | | | | | |
| `plot_synopses` (raw IMDB) | x | | | | | | | |
| `plot_keywords` | x | x | x | x | x | | x | |
| `overall_keywords` | | | x | x | x | x | x | |
| `genres` | | x | x | x | | | | |
| `reception_summary` | | x | x | x | x | | | x |
| `audience_reception_attributes` | | | x | x | | | | x |
| `featured_reviews` (up to 5) | | x | x | x | x | | x | x |
| `maturity_rating` | | | x | | | | | |
| `maturity_reasoning` | | | x | | | | | |
| `parental_guide_items` | | | x | | | | | |

### Key observations

**`featured_reviews` is the largest cost driver.** Each review contains `summary` + `text` (can be 200-500+ tokens per review). Up to 5 reviews are sent to **6 of 8 calls**, meaning roughly 30 review-text transmissions per movie. This is by far the most expensive duplicated input.

**`reception_summary` + `audience_reception_attributes` + `featured_reviews`** form a "reception triple" that appears together in 4 calls (viewer_experience, watch_context, narrative_techniques, reception). Three separate representations of the same underlying signal — what audiences think.

**`plot_keywords` + `overall_keywords`** are both keyword lists sent to overlapping sets of calls. In `source_of_inspiration`, they're already concatenated before sending. Other calls send them separately, which means the model sees partially overlapping keyword sets formatted two different ways.

## 2. Low-Value Inputs Per Generation <a name="low-value-inputs"></a>

### Plot Analysis: `overview` is redundant with `plot_synopsis`

`generate_plot_analysis_metadata` receives both `overview` (marketing blurb) and `plot_synopsis` (the detailed chronological summary generated by Wave 1). The synopsis strictly supersedes the overview — it contains everything the overview has plus far more detail. The overview adds zero incremental information at this point.

**Token impact:** ~30-80 tokens per call wasted.

### Viewer Experience: maturity data serves mostly-skipped sections

`generate_viewer_experience_metadata` receives `maturity_rating`, `maturity_reasoning`, and `parental_guide_items`. These primarily inform the `disturbance_profile` and `sensory_load` sections — both of which are optional (`OptionalViewerExperienceSection` with `should_skip`). For the majority of movies (comedies, dramas, romances), these sections get skipped, but the maturity input tokens are always paid.

`parental_guide_items` is especially verbose: it's a list of `category: severity` pairs formatted into a string. For a typical PG-13 movie this might be 5-8 items adding 40-80 tokens of input for sections that will likely be skipped.

**Token impact:** ~60-120 tokens per call, useful for maybe 20-30% of movies.

### Viewer Experience: `overall_keywords` is noisy

The viewer experience prompt cares about emotional and sensory impact. `overall_keywords` are broad IMDB tags (e.g., "remake", "sequel", "independent film") that rarely carry emotional signal. The plot synopsis and reviews are far more informative for determining how a movie *feels* to watch.

**Token impact:** Variable, but overall_keywords can be 20-100+ tokens of low-signal input.

### Watch Context: `plot_keywords` provides limited signal

Watch context is about WHY/WHEN to watch — "date night movie", "stoned movie", "need a laugh." Plot-specific keywords like "murder investigation" or "time travel" don't inform viewing occasion decisions well. The genres, overview, and reviews are much stronger signals for watch context.

**Token impact:** ~20-80 tokens of low-relevance input.

### Watch Context: `audience_reception_attributes` overlaps with other reception data

This call already receives `reception_summary` and `featured_reviews`. The `audience_reception_attributes` (formatted as "Attribute: X; audience reception: positive/negative") are a structured version of what the reviews and summary already express. This is a third encoding of the same signal in the same call.

**Token impact:** ~30-60 tokens of redundant signal.

### Source of Inspiration: `featured_reviews` rarely discuss production origins

Reviews almost never discuss whether a movie is "based on a novel" or what animation technique was used. Reviewers talk about plot, acting, themes, and emotional impact. The keywords and title are far more informative for source-of-inspiration determination (e.g., a keyword "based on novel" is direct evidence; a review praising "the acting" is not).

The `plot_synopsis` input is marginally useful here (it might reveal the story follows a known real event), but the 5 reviews are likely contributing noise more than signal for this specific task.

**Token impact:** Potentially 500-2000+ tokens of reviews that add near-zero value to this classification task.

## 3. Missing Inputs Per Generation <a name="missing-inputs"></a>

### Watch Context: no `plot_synopsis`

Watch context runs in Wave 1 (parallel with plot_events), so it can't use the generated synopsis. It receives only `overview` — the marketing blurb. This is a significant limitation: a movie's plot details often heavily influence watch context (e.g., a movie about a first date is obviously a "date night movie", but the overview might just say "a romantic comedy about two strangers").

**Trade-off:** Moving watch_context to Wave 2 would add latency (it would need to wait for plot_events). The question is whether the quality gain from having plot_synopsis justifies the latency cost.

### Reception: no `plot_synopsis`

Reception also runs in Wave 1. The prompt asks for what audiences think, which doesn't strictly require plot knowledge — the reviews and reception_summary should be sufficient. This is likely correct as-is.

### Narrative Techniques: no `genres`

Narrative techniques doesn't receive genres. Genres could provide useful context for determining expected narrative patterns (e.g., "mystery" implies information control techniques, "documentary" implies specific POV structures). The model can likely infer this from the synopsis and keywords, but explicit genre labels would be essentially free to include (~5-10 tokens).

## 4. Non-Embedded Output Fields: Are Justifications Worth It? <a name="justification-analysis"></a>

Every metadata type includes justification fields that are generated by the LLM but **never embedded and never stored in the database**. They exist purely as chain-of-thought aids to theoretically improve the quality of sibling output fields.

### Justification field count per call

| Call | Justification Fields | Estimated Output Tokens |
|------|---------------------|------------------------|
| Plot Events | 0 | 0 |
| Plot Analysis | 3-7 (core_concept + themes + lessons + arcs) | ~50-150 |
| Viewer Experience | 8 (one per section) | ~80-160 |
| Watch Context | 4 (one per section) | ~40-80 |
| Narrative Techniques | 11 (one per section) | ~100-220 |
| Production Keywords | 1 | ~10-20 |
| Source of Inspiration | 1 | ~20-40 |
| Reception | 0 | 0 |
| **Total per movie** | **28-32** | **~300-670** |

### The core question: does gpt-5-mini need explicit justification fields?

gpt-5-mini is a **reasoning model** with internal chain-of-thought (reasoning tokens). The model already "thinks through" its analysis before producing output. Explicit justification fields in the output schema were a technique designed for non-reasoning models where forcing the model to write rationale before answers improved accuracy.

With a reasoning model:
- The internal reasoning tokens already serve the chain-of-thought purpose.
- The explicit justification fields may be **redundant** with the model's own reasoning process.
- Every justification field costs output tokens (which are more expensive than input tokens on most models).
- 28-32 justification fields × ~100K movies = significant aggregate cost.

**However, caution is warranted.** The interaction between explicit output justifications and internal reasoning tokens is not well-studied. It's possible that having to *commit* a justification to the output (which gets validated against the structured schema) forces more disciplined reasoning than internal-only CoT. This would need to be tested empirically — generate metadata for a sample batch with and without justification fields and compare quality.

### Other non-embedded output fields

**`PlotAnalysisMetadata.character_arcs[].arc_transformation_description`**: Each character arc has a `character_name`, `arc_transformation_label` (embedded), and `arc_transformation_description` (not embedded but generated). The description is a longer explanation of the arc. Only the label makes it into the embedding. The description costs output tokens but may help the model produce a more accurate label.

**`PlotEventsMetadata.major_characters[].role` and `primary_motivations`**: These ARE included in the `__str__()` output and therefore DO get embedded. These are correctly utilized.

## 5. Redundant Input Signals That Could Be Pre-Consolidated <a name="consolidation"></a>

### Reception data: three signals, one concept

Three separate fields carry audience reception signal:

1. **`reception_summary`** — a concise summary (~50-100 tokens)
2. **`audience_reception_attributes`** — structured attribute+sentiment pairs (~30-80 tokens formatted)
3. **`featured_reviews`** — up to 5 full review texts (~500-2500 tokens)

These are sent independently to 4-6 calls. A pre-consolidated "reception brief" could:
- Merge `reception_summary` and `audience_reception_attributes` into a single condensed paragraph
- Trim `featured_reviews` by extracting only the most information-dense sentences rather than full review texts
- Be generated once and reused across all calls

**Estimated savings:** If reviews average 200 tokens each and 5 are sent to 6 calls, that's ~6000 tokens of review content per movie. A consolidated brief of ~300-500 tokens sent to 6 calls would be ~1800-3000 tokens. Savings: ~3000-4200 input tokens per movie.

**Risk:** Lossy compression of reviews could discard signal that different calls need differently. Plot analysis needs thematic insights from reviews; viewer experience needs emotional language; reception needs quality evaluations. A generic brief might lose task-specific nuances.

**Alternative:** Instead of one brief, create 2-3 task-specific consolidated review excerpts (thematic, emotional, evaluative) and route the relevant one to each call. More complex but preserves signal.

### Keyword lists: two lists, overlapping content

`plot_keywords` and `overall_keywords` are separate IMDB fields, but they overlap significantly. Many calls receive both. Options:
- Merge and deduplicate once, send as a single `keywords` field (as `source_of_inspiration` already does)
- Split into "plot-relevant" and "production/meta-relevant" subsets and only route the relevant subset to each call

**Estimated savings:** Modest — maybe 20-50 tokens per call where both are used. But simplifies the prompt input format.

## 6. Sparse Movies: When Data Isn't Enough <a name="sparse-movies"></a>

Not all movies that pass the IMDB quality filter have rich data. The filter ensures minimum thresholds (IMDB rating exists, directors exist, actors exist, keywords exist), but "exists" doesn't mean "abundant."

### Which inputs are most commonly sparse?

| Input | Sparsity Risk | Impact When Missing |
|-------|--------------|-------------------|
| `plot_synopses` | **High** — many non-blockbusters lack detailed synopses | Plot events has less to synthesize a good plot_summary from, cascading to all Wave 2 |
| `featured_reviews` | **Medium-High** — needs IMDB reviews with sufficient votes | Reception, viewer experience, watch context, narrative techniques all lose their strongest signal |
| `reception_summary` | **Medium** — generated from reviews, may be absent | Reception and viewer experience lose a condensed signal |
| `audience_reception_attributes` | **Medium** — may be empty or contain 1-2 items | Watch context and viewer experience lose structured reception signal |
| `overall_keywords` | **Low-Medium** — usually present but may be sparse (5-10 vs 50+) | Production keywords filtering has less to work with |
| `plot_keywords` | **Low** — usually present in some form | Minor impact — plot synopsis compensates |
| `plot_summaries` | **Medium** — many movies lack user-written summaries | Plot events relies more on overview + model knowledge |

### The cascade problem

Plot events is the foundation. If a movie has no `plot_synopses`, limited `plot_summaries`, and a vague `overview`, the generated `plot_summary` will be thin. Since this `plot_summary` feeds ALL Wave 2 calls as `plot_synopsis`, every downstream generation suffers.

**The worst-case sparse movie:** No plot synopses, no plot summaries, 0-1 featured reviews, no reception summary. At this point, the LLM is working from: title, overview (marketing blurb), genres, keywords, and its own parametric knowledge. For obscure films, that parametric knowledge may be absent or wrong.

### Which generations degrade most for sparse movies?

**Reception metadata — degrades severely.** With no `featured_reviews`, no `reception_summary`, and no `audience_reception_attributes`, there is literally nothing for the model to summarize. It would be fabricating reception data from its parametric knowledge or from the title alone. This is the highest-risk generation for sparse movies.

**Viewer experience — degrades significantly.** The emotional palette, tension, tone sections all lean heavily on reviews and reception data. Without them, the model can infer *something* from plot synopsis and genres (a horror movie is probably tense), but the output will be generic rather than movie-specific. The negation phrases (what the movie is NOT) are especially hard to determine without reception data.

**Watch context — degrades moderately.** Watch scenarios and motivations can be reasonably inferred from genres + overview even without reviews. A horror movie is probably a "halloween movie"; a romance is probably a "date night movie." But the output will be templatic rather than movie-specific.

**Narrative techniques — degrades moderately.** The model can infer structural techniques from plot synopsis alone (if the synopsis mentions flashbacks, it can tag "flashback-driven structure"). But audience_character_perception and meta_techniques sections rely on reviews for evidence.

**Plot analysis — degrades modestly.** Thematic analysis is possible from plot synopsis + genres + keywords. Reviews help but aren't strictly necessary. The model's parametric knowledge of well-known films helps here.

**Production metadata — degrades minimally.** Production keywords is a filtering task on `overall_keywords` (doesn't need rich data). Source of inspiration can often be determined from title + keywords + model knowledge.

### Defining "enough data" thresholds

A movie should be considered too sparse for a given generation if the generation's primary evidence sources are all missing. Suggested minimum thresholds:

| Generation | Minimum Required Inputs |
|-----------|------------------------|
| Plot Events | At least ONE of: `plot_synopses`, `plot_summaries` with meaningful content (>50 words) |
| Reception | At least ONE of: `featured_reviews` (≥1 review), `reception_summary` |
| Viewer Experience | `plot_synopsis` (from Wave 1) AND at least ONE reception signal |
| Watch Context | `overview` or `plot_synopsis` (currently uses overview since Wave 1) |
| Narrative Techniques | `plot_synopsis` with meaningful content (>100 words) |
| Plot Analysis | `plot_synopsis` (from Wave 1) — always available if plot_events succeeded |
| Production | `overall_keywords` (≥5 keywords) — almost always available |

## 7. Partial Embedding: Should Sparse Movies Skip Vector Spaces? <a name="partial-embedding"></a>

Currently, if a metadata generation fails (exception), the metadata is set to `None` and the movie proceeds. The corresponding vector text will either be empty or absent (e.g., `create_reception_vector_text` returns `None` when `reception_metadata` is missing).

### The case FOR partial embedding

- A sparse movie might have excellent plot data but no reviews. Its plot_events, plot_analysis, and narrative_techniques vectors could be high-quality while reception and viewer_experience vectors would be garbage.
- Embedding garbage vectors wastes embedding API cost AND actively harms search quality — a low-quality viewer_experience vector will match queries it shouldn't.
- Skipping vectors for a movie doesn't prevent it from being found. It can still match on its strong vectors. A movie with only 5 of 8 vectors is still searchable via the 5 it has.

### The case AGAINST partial embedding

- Qdrant's named vector architecture handles missing vectors natively — you simply don't include them in the point. The search infrastructure already supports this.
- However, the search scoring pipeline (`db/vector_scoring.py`) may need adjustment. If a movie is missing a vector for a space that has high weight in a given query, the movie effectively gets a 0 in that channel. This could be either correct behavior (the movie genuinely doesn't have enough data to be relevant in that space) or unfair (the movie might actually be a great match but just lacked the data to prove it).

### Recommendation direction

This is worth pursuing. The cost savings are twofold: fewer LLM calls for sparse movies (skip generations that can't produce quality output) AND fewer embedding tokens (skip degenerate vectors). The quality upside is also real — removing noise vectors improves search precision.

The implementation question is whether to make this a hard gate (don't call the LLM at all if inputs are below threshold) or a soft gate (call the LLM but evaluate the output quality and discard if too thin). Hard gates are simpler and save LLM cost; soft gates catch edge cases where the model's parametric knowledge fills the gap.

## 8. Summary of Opportunities <a name="summary"></a>

### High impact (cost savings + quality improvement)

| Opportunity | Mechanism | Estimated Savings |
|------------|-----------|-------------------|
| **Consolidate review input** | Pre-extract key sentences from reviews instead of sending full text to 6 calls | ~3000-4000 input tokens/movie |
| **Skip reception generation for reviewless movies** | Don't call reception LLM when all 3 reception inputs are empty | 1 full LLM call saved for sparse movies |
| **Test removing justification fields** | A/B test metadata quality with and without explicit justification output fields, given gpt-5-mini's internal reasoning | ~300-670 output tokens/movie if removable |

### Medium impact (cost savings, minimal quality risk)

| Opportunity | Mechanism | Estimated Savings |
|------------|-----------|-------------------|
| **Drop `overview` from Wave 2 calls** | Plot synopsis already supersedes it for plot_analysis | ~30-80 input tokens/movie |
| **Drop `featured_reviews` from source_of_inspiration** | Reviews rarely discuss production origins | ~500-2000 input tokens/movie |
| **Drop `audience_reception_attributes` from watch_context** | Already receiving reception_summary + featured_reviews | ~30-60 input tokens/movie |
| **Drop `parental_guide_items` from viewer_experience** | Only useful for disturbance_profile/sensory_load (often skipped); maturity_rating + maturity_reasoning already convey the essential signal | ~40-80 input tokens/movie |
| **Merge keyword lists** | Deduplicate plot_keywords + overall_keywords before sending | ~20-50 input tokens per call where both appear |

### Architectural (requires design work)

| Opportunity | Mechanism | Estimated Savings |
|------------|-----------|-------------------|
| **Partial embedding for sparse movies** | Define per-generation data thresholds; skip LLM + embedding for spaces below threshold | 1-4 LLM calls + embedding tokens for sparse movies |
| **Move watch_context to Wave 2** | Give it access to plot_synopsis | Net zero cost, quality improvement for watch context output |
| **Add genres to narrative_techniques** | Essentially free input, helps ground structural analysis | ~5-10 input tokens, potential quality improvement |

### Needs empirical testing

| Question | How to Test |
|----------|------------|
| Do justification fields improve output quality for gpt-5-mini? | Generate metadata for 50-100 movies with/without justifications, blind-evaluate quality |
| Does `overall_keywords` improve viewer_experience quality? | Generate with/without, compare term relevance |
| Does `plot_keywords` improve watch_context quality? | Generate with/without, compare term relevance |
| What's the quality threshold for "too sparse"? | Sample movies at various data density levels, evaluate metadata quality to find the cliff |
