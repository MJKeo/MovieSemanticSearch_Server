## Concrete typed LLM output when format parity is required
**Observed:** When designing the step 3 semantic endpoint, weighed freeform `query_text` strings vs. concrete per-space Pydantic objects matching ingestion-side embedding shape. Chose concrete objects because freeform strings make the LLM the only enforcement of format parity — a silent drift risk. Concrete objects wrapped in discriminated unions (`Literal[space]` on each wrapper) bind the format at the schema layer; cross-side format drift becomes structurally impossible.
**Proposed convention:** When LLM output must match a format produced elsewhere in the system (e.g., ingestion-side embedding text that query-side vectors need to align with), emit concrete typed objects, not freeform strings. The Pydantic schema should mirror the target format's shape closely enough that a shared or equivalent `embedding_text()` / formatter method on both sides produces byte-identical output.

## Duplicate embedding/format logic deliberately over factoring into shared helpers
**Observed:** When designing query-side `*Body` classes that mirror ingestion-side `*Output.embedding_text()`, considered factoring the shared formatter into a helper. Chose to duplicate instead, on the reasoning that duplication surfaces divergence in code review whereas a shared helper would silently change both sides' behavior on refactor.
**Proposed convention:** When two sides of the system must produce identical formatted output but evolve independently (different data sources, different validation needs), duplicate the formatting logic rather than factoring into a shared helper. Document the duplication intent in-file. This converts silent drift into a code-review-visible diff.

## Query-side schemas shouldn't mechanically replicate ingestion-side data-hygiene guards
**Observed:** `ProductionBody` initially replicated ingestion's `filming_locations[:3]` truncation. User corrected: the cap is ingestion-side defense against scraped noise; on the query side the LLM emits only intentional locations, so silently truncating a 4th entry discards real signal. The underlying principle: a guard that's correct on one side can be wrong on the other when the data sources and failure modes differ.
**Proposed convention:** When mirroring schemas across pipeline stages (ingestion vs. query, raw vs. synthesized), evaluate each guard/transformation's reason-for-being before carrying it across. Guards against upstream data noise belong only on the ingestion side. Guards enforcing invariants the downstream consumer depends on belong on both sides. Don't copy the guard just because the shape is the same.

## Edge-case guards belong at the narrowest caller, not in shared helpers
**Observed:** When fixing a lone-hyphen token leaking into `studio_token`, scoped the guard to `tokenize_company_string` (studio-only wrapper) rather than modifying the shared `tokenize_title_phrase`. Motivation: title tokenization has been stable and the same "bare `-` → drop" transform hasn't been validated for titles; changing the shared helper would silently alter every existing lex.lexical_dictionary consumer. User accepted the narrower scope without pushback and the docstring explicitly called it out ("without altering the shared title tokenizer's behavior"). Same instinct surfaced in the module docstring ("The shared `normalize_string` in helpers.py is intentionally left alone — rewriting it globally would invalidate every existing lex.lexical_dictionary row").
**Proposed convention:** When a defensive guard or transform is correct for one caller but has not been verified to be correct for every caller of a shared helper, apply the guard at the narrow caller — a wrapper, post-filter, or specialized helper — not by editing the shared helper in place. Shared helpers' invariants are load-bearing for every downstream index and cache; change them only when the new behavior is deliberately required across every caller, and when the cost of a follow-up rebuild of all derived data is understood.

## Preserve source columns alongside derived search arrays
**Observed:** Both the studio resolver (keeps raw `production_company.canonical_string` strings alongside the derived `movie.brand_ids INT[]` stamp) and the new franchise design (keeps raw `lineage` / `shared_universe` / `recognized_subgroups` TEXT columns alongside the derived `franchise_name_entry_ids INT[]` / `subgroup_entry_ids INT[]` arrays) follow the same pattern: search-time access uses the denormalized ID arrays, but the original LLM-authored TEXT columns remain on the row. User explicitly directed this for franchise: "We can keep the older columns for debugging / easily seeing what is what for a given movie, but for searching it should be one list."
**Proposed convention:** When introducing a denormalized / projected representation for search (GIN-indexed entry-id arrays, stamped brand arrays, token-index entries), do not drop the original source columns. Keep the canonical human-readable TEXT form on the row. Reasons: (1) debugging — "what did the LLM actually write?" must be answerable without re-running generation; (2) rebuildability — stoplist/normalizer changes let you regenerate the index from the source columns without LLM re-generation; (3) non-search consumers (movie cards, analytics, evals) still read the human-readable form. The projected columns are derivable; the source is not.

## Pydantic validator scope: quality fixes + presence, not cross-field dependencies
**Observed:** During the franchise query-side cutover the user directed removal of the `recognized_subgroups requires lineage_or_universe_names` cross-field validator on `FranchiseQuerySpec`, because a spec like "trilogies" (subgroup-only, no named franchise) is a legitimate standalone query. User stated: "A franchise spec should be valid with any field configs. All we should do in _validate() is small data quality fixes (like deduping structural flags) and checking that at least one piece of data is present." The executor then took responsibility for empty-axis handling — not the schema validator.
**Proposed convention:** Pydantic model validators should be limited to (a) data-quality fixes (dedup, order preservation, trimming) and (b) presence guards ("at least one axis populated" when the model has no intrinsic use without any field). Avoid cross-field dependency rules like "X requires Y" unless the combination is structurally meaningless — a pattern like "X's meaning depends on Y being set" usually encodes a usage convention, not an invariant, and blocks legitimate standalone uses. Let the executor enforce axis-combination semantics so standalone axes remain composable.

## Stopword / stoplist filtering lives inside the shared tokenizer, not the executor
**Observed:** Award name resolution (`AWARD_STOPLIST` inside `tokenize_award_string`) and the newly-landed franchise resolver (`FRANCHISE_STOPLIST` inside `tokenize_franchise_string`) both apply their stoplists inside the shared tokenizer that ingest and query call, rather than post-filtering in the executor. Studio omits a stoplist entirely and relies on a DF ceiling. User's open-question #1 response confirmed the pattern for franchise: "get the list of tokens from the shared helper method then remove any that are present in the stopwords set, and only then do we do batch token ID fetching." This means the executor never sees a stopword, so there is no place for a symmetry drift to sneak in.
**Proposed convention:** For freeform token-index resolvers (any domain where the same tokenizer runs at ingest and query), apply the stopword drop inside the shared tokenizer function — not as a post-filter in the executor. This keeps the "ingest and query share one tokenizer" invariant observable from one file instead of split across the ingest path, the query executor, and any future callers. If a stoplist needs to be curated over time (via a DF materialized view diagnostic), edits happen in one place and both sides update symmetrically without any code change in the executor.

## Branch SQL query shape to match the active predicate set
**Observed:** First draft of `fetch_franchise_movie_ids` always drove from `movie_card` (100K rows) and optionally LEFT JOINed `movie_franchise_metadata` (smaller) when structural predicates were active. The review flagged that for structural-only specs ("spinoffs", "sequels with no franchise") the v1 code scanned mfm directly, whereas the new v2 code forced an unnecessary join. Post-fix: the helper now picks between three from-clauses: structural-only → `mfm` directly; array-only → `mc`; mixed → `mc LEFT JOIN mfm`. `restrict_movie_ids` binds to the driving table's `movie_id` column per branch. User approved.
**Proposed convention:** When a helper accepts multiple independently-populated predicate groups that live on different tables (and those tables differ materially in row count), branch the from-clause to the smallest driving table for each predicate shape rather than using one universal shape pessimized for the rare cases. The branching is cheap — a handful of Python if-statements — and avoids forcing an extra index traversal or join on every call. Prefer driving from the smaller predicate-bearing table; only pull in wider tables when their predicates are actually active.

## Small freeform corpora prefer query-only stoplists over bilateral ones
**Observed:** Award name resolution (~585 distinct normalized entries) diverged from the bilateral-stoplist pattern used by franchise (~5.8K entries). The award ingest tokenizer (`tokenize_award_string`) emits every token; the query side wraps it in `tokenize_award_string_for_query` which additionally drops `AWARD_QUERY_STOPLIST`. User's explicit rationale during plan discussion: "We will create [the stoplist] for the query side once we run ingestion and see the final results" — and later confirmed the droplist be revisable without re-ingest. Token-table cost is ~2.2k rows regardless of stoplist, so keeping every token preserves the ability to re-curate the droplist from the DF materialized view without re-running ingest. Existing convention #7 ("stopword filtering lives inside the shared tokenizer") still holds at the layer-placement level — both ingest and query tokenizers live in the same module and share the same `normalize_award_string` — the novelty is the intentional one-step asymmetry.
**Proposed convention:** For freeform token-index resolvers where the entry corpus is small (roughly <~1K distinct entries), prefer a query-only stoplist: ingest writes every token to the posting table, query wraps the ingest tokenizer with an additional stoplist drop. This preserves the ability to revise the stoplist by config change alone without triggering a re-ingest. Keep both tokenizers in the same module file so the asymmetry is visible in one place and the shared normalization is still the single source of truth. For larger corpora (≥~5K entries where posting-table bloat matters), the bilateral pattern (stoplist in the shared tokenizer, applied symmetrically) remains appropriate — the franchise path is the reference example.

## Idempotent one-shot migrations that reuse existing ingest writers
**Observed:** For the franchise lineage/universe column split, the migration at `db/migrate_split_franchise_columns.py` combined two patterns: (1) schema statements used `DROP INDEX/COLUMN IF EXISTS` + `ADD COLUMN IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS` so the ALTERs are safe to re-run on both fresh and partially-migrated DBs; (2) the backfill loop called the existing `write_franchise_data` helper — which was already idempotent via its `ON CONFLICT DO NOTHING` upserts against `lex.franchise_entry` / `lex.franchise_token` — rather than re-implementing the write logic. The alternative (destructive volume wipe + full re-ingest) was explicitly rejected by the user. The migration finished by calling `update_movie_card_franchise_ids` to stamp only the two new columns; unrelated `movie_card` columns and rows in other tables were left untouched.
**Proposed convention:** One-shot Postgres migrations should: (a) use `IF EXISTS` / `IF NOT EXISTS` on every schema statement so the script is idempotent and survives partial runs; (b) reuse existing idempotent ingest helpers for any data recomputation rather than re-implementing write logic — the helpers already encode the normalization, entry-id resolution, and ON CONFLICT behavior; (c) touch only the columns the schema change actually affects, leaving unrelated derived data intact; (d) cap concurrency via `asyncio.Semaphore` below the pool's `max_size` when parallelizing the backfill. Do NOT default to dropping the Postgres volume and re-ingesting — only do that when the source of truth for the migration is gone or the change is genuinely table-wide.

## Verdict unification for variable-cardinality decisions
**Observed:** The Step 2A prompt originally had two verdicts that tried to separately handle "translate this phrase to one broader thing" (`best_guess`) vs "split this phrase into several retrieval targets." In practice the distinction collapsed into "how many output atoms does this phrase produce" — one or more — and the pre-classification gate caused under-expansion (phrases that should have produced multiple atoms were forced through the single-atom path because the verdict was picked before the content was written). Fix: replaced with a single `interpret -> 1+ atoms` verdict where cardinality reveals itself from the phrase; picked up in the rewrite and validated on the probe.
**Proposed convention:** When two LLM verdicts distinguish only by output count (one thing vs several things), collapse them into a single verdict with variable cardinality. Do not force the model to pre-classify "which kind of output is this" before it has written the output — let the content determine its own shape. Name the verdict for the action ("interpret", "extract"), not the count. Reserve distinct verdicts for genuinely different operations (e.g., interpret vs fold_into vs filler), not for cardinality variants of the same operation.

## Bidirectional coverage validators on Pydantic partition models
**Observed:** `Step2AResponse`'s `validate_partition_completeness` enforces two invariants against the committed `inventory` list: (1) every `slot.scope` entry must appear in `inventory` (slots cannot cite phrases that were never committed), AND (2) every `inventory` entry must appear in some `slot.scope` (committed phrases cannot silently fall on the floor). The first direction catches referential errors; the second catches content loss. An earlier review flagged that the validator only enforced direction (1), leaving (2) to prompt discipline. Adding (2) closed a silent-drop failure mode that no prompt text could have prevented.
**Proposed convention:** When a Pydantic model represents a partition of an enumerated source (slots partitioning an inventory, concepts partitioning required ingredients, categories partitioning terms), the validator must enforce coverage in BOTH directions — every child reference must exist in the source AND every source entry must be referenced by some child. Single-direction validators admit silent drops that prompt discipline alone cannot catch. Keep the check in `model_validator(mode="after")`; fail loudly with a message that names the orphaned entries.

## Grouping rules need a sub-dimension clause when the parent category has internal structure
**Observed:** Step 2A's fusion rule was first written as "same retrieval family + ranking-style qualification → fuse." The probe surfaced an observed inconsistency where same-family atoms on DIFFERENT sub-attributes (metadata.popularity + metadata.reception, or keyword.character_type + keyword.plot_archetype) sometimes fused and sometimes split across runs. Root cause: the rule left ambiguity about whether "same family" meant same top-level family (metadata vs keyword) or same sub-dimension within the family. Fix: tightened to "same family + same sub-dimension within multi-attribute families (metadata's 10 attributes, keyword's 7 categories) + ranking-style." Semantic remained a single fusion unit because its vector spaces can legitimately be queried together in one slot. Post-fix runs consistently split cross-attribute metadata atoms.
**Proposed convention:** When authoring a grouping/fusion rule over a categorical dimension that itself has internal structure (sub-categories, sub-attributes, sub-types), the rule must commit explicitly to either the parent level or the sub-level. "Same category" alone is ambiguous when the category has sub-structure, and the model will resolve the ambiguity inconsistently across runs. If the downstream consumer operates at the sub-level (one metadata expression = one column, one keyword expression = one category), the rule must require same sub-level. If the downstream consumer treats the parent as one unit (semantic's vector spaces combine in one slot), name that explicitly. Observed run-to-run inconsistency on such rules is the diagnostic signal that the sub-dimension clause is missing.

## Enum vocabularies for LLM structured output carry prompt-ready descriptions as enum attributes
**Observed:** `CategoryName` (the 31-member step-2 categorization vocabulary) was moved into `schemas/enums.py` with a per-member `description` attribute, using the tuple-constructor `__new__` pattern already in use by `NarrativeStructureTag`, `LineagePosition`, and several concept-tag enums. `search_v2/step_2.py` then renders the taxonomy section of its system prompt programmatically via `_build_category_taxonomy_section()`, iterating `CategoryName` and emitting `Cat: <value>\n  <description>`. User directed this shape explicitly: "Each category enum value should have a regular value that's just its name and a description value that contains all the necessary context we'd want to inject into this LLM (so attributes contained, concepts covered, questions answered, but not routing logic). Then add those changes to the preprocessor." Descriptions are intentionally concept-only — routing, mechanism, and endpoint details belong to the dispatcher, not the LLM's vocabulary.
**Proposed convention:** When an enum is both the structured-output vocabulary for an LLM (a field like `category_name`, `tag`, `route`, `family`) AND the LLM needs per-value concept descriptions to pick correctly, carry the description on the enum itself via the tuple-constructor pattern (see `NarrativeStructureTag` for the reference shape). Render the prompt's taxonomy section programmatically from the enum rather than maintaining a parallel prompt string. This converts "prompt text drifts from schema" into a structural impossibility — one edit updates both surfaces. Keep descriptions concept-only (what the value covers); routing/dispatch details live with the dispatcher, not the enum.

## Persist enum ID stability over alphabetical order when growing taxonomies
**Observed:** When adding `LIVE_ACTION` to `OverallKeyword` (225-member genre/sub-genre taxonomy persisted as `INT[]` in `public.movie_card.keyword_ids` with a GIN index), the user explicitly directed: "NO SHIFTING EXISTING ID NUMBERS EVEN IF IT MEANS WE NO LONGER HAVE PERFECT ALPHABETICAL ORDER." The new member was appended at the end as `(226, "Live Action", ...)` out of alphabetical position. Renumbering would have silently invalidated every downstream consumer that stored keyword_ids by value (movie_card rows, Qdrant payloads, posting lists) — the cost of a follow-up backfill across 100K+ rows far outweighs the readability benefit of keeping the enum alphabetized.
**Proposed convention:** When extending a numerically-ID'd enum whose IDs are persisted in any downstream store (DB columns, search indexes, posting lists, exported JSON), always append new members at the end with a fresh ID. Never renumber existing members to preserve alphabetical or logical ordering, even if that would be cleaner for readability. Add a short comment at the insertion point noting the ID is deliberately out of order to preserve stability. The alphabetical/logical ordering can be recovered via a sorted iteration helper if needed for presentation, but the on-disk IDs must remain stable across schema growth.

## Per-sub-task error isolation in async orchestrators
**Observed:** Built `search_v2/steps_0_2_orchestrator.py` to coordinate steps 0-2. User defined error discipline as "treat each flow as independent": step 0 failure fatal, step 1 failure bounded to the standard flow only, step 2 failure bounded to the individual branch. Implemented via `asyncio.gather(..., return_exceptions=True)` + per-helper `try/except Exception` that returns a result object with `response=None, error=repr(exc)` so callers can distinguish failure from absence. Sibling tasks and unrelated flows keep running.
**Proposed convention:** For any async orchestrator that fans out into independent downstream work, isolate failure domains at the boundary: use `asyncio.gather(return_exceptions=True)` for the fan-out, wrap each sub-task's execution in a helper that catches `Exception` (not `BaseException` — cancellation still propagates) and returns a structured result carrying either the response or the error. One sub-task's failure must not cancel its siblings or blow up unrelated parallel flows. Only raise at the orchestrator level when the failure genuinely prevents downstream dispatch (e.g., routing/classification steps whose output every later step depends on).

## Nullable stats for failed sub-operations
**Observed:** Code review of the steps 0-2 orchestrator flagged that `Step2Branch` reported `input_tokens=0, output_tokens=0, elapsed=0.0` on failed branches — truthful but misleading, because naive aggregators summing across branches would silently undercount a failure as a zero-cost success. User directed fixing this by changing the fields to `int | None` / `float | None` with `None` on failure.
**Proposed convention:** When a result type reports stats about a sub-operation that may have failed (token counts, elapsed time, row counts, hit ratios), use `None` rather than a zero sentinel for failed cases. `None` forces callers to acknowledge the failure state (either filter or explicitly treat as zero), while `0` silently blends into totals. Applies any time the data carrier also carries an error field — zero is a measurement, not an absence.

## Dispatch-routing metadata on vocabulary enums, not parallel maps
**Observed:** `CategoryName` (the 32-member step-2 categorization vocabulary) already carried a `.description` attribute for the LLM prompt; this session added `.endpoints: tuple[EndpointRoute, ...]` to every member so the taxonomy itself names which retrieval endpoints each category may dispatch to, in priority order. The alternative — keeping a separate `CATEGORY_TO_ENDPOINTS: dict[CategoryName, tuple[EndpointRoute, ...]]` module-level map — was rejected because it would let the vocabulary and its dispatch surface drift independently (a new enum value could be added without a dispatch entry, and vice versa). User directed: each category value gains an endpoints list on the enum itself. Same tuple-constructor pattern as `.description`. The user also noted priority ordering applies to all members even though only tiered categories strictly depend on it — "good principle overall."
**Proposed convention:** When an enum is the authoritative vocabulary for a dispatch decision (category → endpoint, tag → handler, route → service), carry the dispatch target(s) as an enum attribute via the tuple-constructor pattern — not in a parallel dict keyed by the enum. Use a tuple of targets rather than a single target even when most members currently dispatch to one place, and order it by priority so downstream code can treat the list uniformly. This converts "a new enum value without a dispatch entry" from a runtime KeyError into a structural impossibility (the `__new__` signature refuses members missing the field). Extends the existing "enum vocabularies carry prompt-ready descriptions" convention to the routing surface: both LLM-facing descriptions and code-facing dispatch live on the vocabulary itself.

## Failure logs include only the fields the active branch consulted
**Observed:** The unified semantic executor originally logged `primary_vector=...` and `spaces=...` on every retry/error, regardless of whether the request was dealbreaker (uses primary_vector, ignores the full list) or preference (uses the full list, ignores primary_vector). User pushed back during code review: "Only log what's actually used and relevant." Fixed by computing `log_context` once per branch before the retry loop — identification gets `primary_vector=...`, reranking gets `spaces=...` — and reusing that single string in both the warning and error log calls. No union-of-all-fields in either message.
**Proposed convention:** When a function dispatches on a mode/role flag and each mode consumes a different subset of the input fields, failure logs must include only the fields the active mode actually used. Kitchen-sink logs (every possibly-relevant field, regardless of branch) mislead operators into considering fields that the failure did not involve — especially when a field is explicitly documented as ignored in the active mode. Compute the log context once at the top of the function once the mode is known, then reuse the same string for all log sites in that call.

## Pure scoring helpers; dealbreaker-vs-preference branching at the executor
**Observed:** When aligning stage-3 dealbreaker scores to the [0.5, 1.0] floor, the user confirmed the compression should apply **only** to the dealbreaker path, not the preference path (which keeps raw [0, 1] gradient for ranking resolution). Rather than adding a `dealbreaker: bool` parameter to each scoring helper (`_score_release_date`, `_score_popularity`, `_score_reception`, `_score_from_count`, `_threshold_flatten`, etc.), the implementation kept those helpers pure single-responsibility functions returning raw [0, 1] scores and branched on `restrict_to_movie_ids is None` at the executor's call site to apply `compress_to_dealbreaker_floor(raw)`. The `_threshold_flatten` exception (modified in place) was justified because exploration confirmed it was already only reachable from dealbreaker code paths — preference used a different path (`_weighted_cosine_score`). Trending's compression similarly happened in the executor rather than in `compute_trending_score` (which writes to Redis and is shared with preference).
**Proposed convention:** When an endpoint has dual-mode behavior (dealbreaker vs preference) that requires different score shaping, keep the pure scoring helpers returning raw [0, 1] values and branch at the executor based on `restrict_to_movie_ids is None`. Don't parameterize the helpers with a mode flag — that couples every scorer to the branching logic and makes it harder to reason about what a scorer returns in isolation. Only modify a scoring helper in place when it's provably unreachable from one of the two paths (single-caller check via grep, confirmed in the module docstring). Bonus: the affine compression itself (`0.5 + 0.5 * raw`) is a shared utility in `search_v2/stage_3/result_helpers.py:compress_to_dealbreaker_floor` — new endpoints needing the same floor should reuse it, not reinvent.

## Pydantic v2 preserves base-class field order; subclass redeclaration does NOT reorder
**Observed:** When reordering `action_role / parameters / polarity` on `EndpointParameters` subclasses, I first assumed that redeclaring base-class fields in the subclass would move them to the subclass's declaration order. It does not — Pydantic v2 preserves base-class field positions regardless of redeclaration (the subclass's new fields append after the inherited ones). Verified empirically via `model_json_schema()["properties"]`: the order came out `action_role, polarity, parameters` even when subclasses declared them in `action_role, parameters, polarity` order. The fix was to strip the fields off the base entirely and declare all three in each subclass. The empty base still serves its load-bearing purpose (isinstance routing for `HandlerResult.preference_specs`), and a `__pydantic_init_subclass__` hook enforces presence + order so the invariant can't silently rot.
**Proposed convention:** When LLM-facing JSON schema property order matters (attention-anchoring, avoiding premature commitment, e.g. retrospective classification fields that must come after content), verify the order via `model_json_schema()["properties"]` before shipping. Do NOT assume that redeclaring a base-class field in a subclass moves its position — it doesn't. If a field needs to appear at a specific position in the subclass's schema, declare it in the subclass (strip from the base if necessary); the base class then becomes a marker/contract holder. Add a `__pydantic_init_subclass__` hook (NOT `__init_subclass__` — the latter fires before `model_fields` is populated) to enforce presence and order invariants across all subclasses.

## LLM-facing field descriptions stay inside the LLM's grounded vocabulary
**Observed:** Two iterations of field description writing in this session landed in very different places. First pass wrote `action_role` and `polarity` descriptions with system-architecture vocabulary — "candidate pool", "orchestrator", "already-decided pool", "subtract or downrank matches", "routing signal". User pushed back directly: "You just added a ton of code-specific context to the LLM, how does that help it at all? I'd guess it'd distract since we don't give it knowledge on the broader system as a whole." Second pass rewrote both descriptions using only concepts graspable from the handler's visible prompt context — "drops out entirely" / "still qualifies, just with a lower score" / "describe the target concept directly, not its inverse" — plus worked example quadrants. Same rule applied to the `shared_vocabulary.md` prompt chunk and enum values (`filter / trait` rather than `candidate_identification / candidate_reranking`).
**Proposed convention:** When writing `Field(description=...)` text for Pydantic models used as LLM structured-output schemas, or any text that ends up in the LLM's prompt via JSON Schema, every concept in the description must be graspable from the LLM's visible prompt context alone. Treat architecture-layer vocabulary (orchestrator, candidate pool, inclusion_candidates, routing, downstream handlers, etc.) as unavailable — those concepts live only in developer-facing code comments and docs that never reach the model. Rewrite in user-intent language with concrete worked examples covering the main failure-mode quadrants. Developer comments inside the .py file are still fine — this rule governs only the strings that propagate into the JSON schema via `Field(description=...)`, enum values, and enum-attribute descriptions. Audit by reading the description as if you'd never seen the codebase and checking whether every concept is either (a) self-defined inline, (b) a common-English word, or (c) already defined in the shared vocabulary block.

## Enum values and field names for LLM-facing vocabulary should visibly convey the underlying asymmetry
**Observed:** Naming the hard/soft classification enum went through several rejected pairs before landing on `MatchMode: filter | trait`. Key iterations: `ActionRole: candidate_identification | candidate_reranking` (rejected — system-speak, ungroundable from the LLM's prompt); `strictness: requirement | preference` (rejected — "requirement" collides with the `requirement_aspects` reasoning field already in play); `constraint / preference` (rejected — doesn't visibly convey that one side is a binary yes/no test while the other is a descriptive attribute). The user's framing on why `filter / trait` won: "One is a hard, binary yes-no type of question that lets us reduce the candidate pool while the other is just descriptive traits that qualify those selected movies" — the names now make that asymmetry legible at the name level, which means the LLM sees it structurally rather than having to derive it from descriptions.
**Proposed convention:** When naming enum values or field names for LLM-facing vocabulary, two rules stack: (1) check for collisions with existing codebase vocabulary (grep for near-root synonyms in schemas/, prompts/, reasoning-field names) and surface collisions as part of the proposal; (2) if the underlying distinction between two values is structurally asymmetric (binary test vs gradient, hard-gate vs soft-nudge, filter vs descriptor), pick names that convey that asymmetry at the name level rather than symmetric-sounding pairs. Symmetric-sounding pairs ("requirement / preference", "constraint / preference", "identification / reranking") ask the LLM to recover the asymmetry from descriptions; asymmetric-named pairs ("filter / trait") let the model pattern-match on the names directly. The convention is an extension of "LLM-facing descriptions stay inside the LLM's grounded vocabulary" applied at the level of identifiers, not just prose.

## UNKNOWN sentinel for enums whose IDs persist in typed Postgres columns
**Observed:** When designing the `ReleaseFormat` enum (mapped from IMDB's `titleType.id` and persisted in `movie_card.release_format SMALLINT NOT NULL DEFAULT 0`), the user explicitly directed including an `UNKNOWN=0` member to flag inputs outside the supported set ("movie / tvMovie / short / video"). The standard "Safe lookups for external data" convention (return `None` for unrecognized inputs) doesn't fit when the consumer is a NOT NULL typed column — `None` can't land in the column, and using `NULL` would lose the distinction between "out of scope" and "not yet computed." The UNKNOWN sentinel doubles as the column default so freshly-added rows are flagged until backfilled, and as a queryable audit handle for unsupported content that slipped past upstream gates. The helper `release_format_id_for_imdb_type(imdb_type: str | None) -> int` returns 0 (UNKNOWN) for both `None` and unrecognized strings.
**Proposed convention:** When an enum's int IDs are persisted in a typed Postgres column (especially NOT NULL with a default), include an explicit `UNKNOWN=0` (or equivalent sentinel) member rather than relying on the standard return-`None` lookup pattern. Two reasons: (1) the column default can be 0, so an `ALTER TABLE ADD COLUMN ... DEFAULT 0` materializes every existing row in a known-bad state until backfill flips them — safer than NULL, which conflates "no data" with "intentional null"; (2) the sentinel is queryable (`WHERE release_format = 0` or `GROUP BY release_format`), giving operators an audit handle for content that slipped past upstream filters. Pair the sentinel with a typed helper (`x_id_for_external_string(s: str | None) -> int`) that maps both `None` and unrecognized strings to the sentinel int. The "Safe lookups return `None`" convention still applies when the consumer is in-process Python that can branch on `None`; this pattern is its complement for typed-column persistence.

**Proposed convention:** When writing LLM-facing instructions for natural-language interpretation tasks (parsing, classification, salience weighting, attribute assignment), prefer principle statements paired with calibration examples over codified decision rules. ## Trust hierarchy in multi-step LLM pipelines: structural decisions authoritative, descriptive content advisory
**Observed:** Step 1 of the new query-understanding pipeline produces a prose `holistic_read` that contains both structural markings (this is an anchor / qualifier / kept-whole / hybrid) and descriptive content (what aspects of Inception likely matter, what "with my mom" implies). Coverage testing across ~25 queries surfaced an asymmetric reliability pattern: structural markings are reliable across simple, dense, and edge-case queries; descriptive content has residual failures (named-work expansion, anchor-pattern hallucination, use-case glossing). The downstream pipeline design (Steps 2-5) was scoped explicitly around this asymmetry — Step 2 trusts Step 1's structural calls (loose, can override with reasoning); Step 4 trusts the structural anchor classification but regenerates parametric expansion fresh from world knowledge rather than anchoring on Step 1's descriptive guesses. Trust hierarchy was named explicitly in the planning doc as a cross-cutting principle.
**Proposed convention:** When designing a multi-step pipeline where one LLM step's output feeds another's prompt, evaluate the upstream step's outputs against an axis of "what dimensions are reliable vs. what dimensions have residual failures." Reliable dimensions (typically: structural classifications, presence/absence judgments, role naming, kept-whole flags) become *authoritative* — downstream prompts treat them as decisions, not suggestions. Unreliable dimensions (typically: open-ended descriptions, expansion of named references, dimension enumeration) become *advisory* — downstream prompts may use them as informational context but should regenerate the underlying content fresh rather than anchoring on the upstream guess. Surface this asymmetry explicitly in downstream prompts ("trust the anchor classification; regenerate the expansion fresh") rather than letting the LLM infer it. Eval the upstream step's reliability pattern empirically on a coverage battery before committing to the trust hierarchy — don't assume what's reliable.

## Loose trust between adjacent LLM steps when upstream has any failure rate
**Observed:** When designing Step 2 of the pipeline (which consumes Step 1's holistic read), the open question was whether Step 2 should treat Step 1's structural markings as strict (must produce atoms consistent with them — overrides forbidden) or loose (strong defaults but Step 2 can override with explicit reasoning). User chose loose. Rationale: Step 1 has occasional descriptive failures and the structural markings, while reliable in aggregate, are not infallible — particularly on queries with anchor-pattern hallucination. Strict trust would let Step 1 errors cascade with no downstream check; loose trust preserves robustness at the cost of some redundant analysis. The convention pattern: default to consistency with the upstream step, but allow overrides when reasoning is explicit, and require the disagreement to be surfaced in the override's reasoning field.
**Proposed convention:** When designing a downstream LLM step that consumes an upstream LLM step's structured output, default to "loose trust": the upstream's calls are strong defaults but the downstream step is allowed to override them with explicit reasoning. Do not design for strict trust (no overrides) unless the upstream step has been measured at near-zero error rate on the relevant decisions. The downstream prompt should: (a) instruct the model to default to consistency with the upstream call, (b) require any override to surface the disagreement in a reasoning field rather than silently diverging, (c) reserve overrides for genuine reasoning, not preference. This converts cascading-error risk into a tractable per-step audit trail without redoing the upstream analysis from scratch.

## Defer compression to the latest layer that can do it cleanly in multi-stage LLM decompositions
**Observed:** Step 3's prior prompt allowed dimensions to fold "two aspects sharing one searchable check" into one dimension. v10 batch showed this caused silent aspect drops (Wes Anderson's "meticulous production design", "darker tone"'s "lack of whimsy") because the call layer couldn't recover information already merged at the dimension layer. User explicitly directed: "should look at ALL aspects even if it seems redundant, since we re-merge in the final category calls list" and "no silent drops, no aspect-merging at this layer. Compression happens at category_calls." Removing the dimension-layer collapse rule and adding "translate every aspect" framing dropped silent drops from 55 to 34 (-38%) without bloating call counts (only +4 across 86 traits). Same pattern surfaced in earlier work: q34 hidden gem's "FINANCIAL_SCALE leakage" was treated as a bug under the v10 CLEAN-FIT rule but was actually correct decomposition that the over-eager intermediate-layer compression was suppressing.
**Proposed convention:** When designing multi-step LLM transformations where data could legitimately compress at multiple layers (e.g., aspects → dimensions → calls; atoms → traits → routings; entities → expressions → endpoints), defer compression to the LATEST layer that can do it cleanly. Earlier layers must preserve maximum granularity even when the redundancy looks obvious. Pre-merging at intermediate layers creates irrecoverable silent drops because downstream layers cannot recover information that was already collapsed. The merge logic at the latest layer (e.g., same-category dimensions collapsing into one multi-expression call) handles the compression naturally without losing axes. Diagnostic signal that compression is happening too early: aspects/inputs at layer N have no traceable representation at layer N+1 (silent drops) — a structural test the prompt should explicitly ban. Companion to existing per-stage commit conventions: at each layer, the question is "can the next layer recover what I'm about to merge?" — if no, don't merge.

## `_ENDPOINT_PROMPTLESS` membership pairs with a handler short-circuit — never one without the other
**Observed:** During the media_type endpoint implementation, code review caught that the LLM-handler stack has a structural invariant the codebase doesn't make obvious: `prompt_builder._ENDPOINT_PROMPTLESS` (which excludes a route from prompt eager-loading) and `handler.run_handler`'s explicit category short-circuits (which return empty `HandlerResult` before the LLM codepath runs) MUST agree. If a category routes only to PROMPTLESS endpoints, `build_system_prompt` raises ValueError with "no LLM-wrapper endpoints" — so the handler must short-circuit that category before reaching the LLM. Conversely, removing a short-circuit without authoring the prompt + removing from PROMPTLESS leaves the handler unreachable but no longer protected. In this session both transitions surfaced explicitly: adding to PROMPTLESS required adding the short-circuit; later removing the short-circuit required first authoring the prompt + removing from PROMPTLESS; pivoting back required both reversed.
**Proposed convention:** Treat `prompt_builder._ENDPOINT_PROMPTLESS` membership and `handler.run_handler`'s per-category short-circuits as a paired invariant. Whenever a route is added to or removed from PROMPTLESS, audit `handler.run_handler` for the corresponding short-circuit. Whenever a category short-circuit is added or removed, audit PROMPTLESS for the matching state. The comment block above the frozenset should explicitly enumerate which categories are short-circuited and why, so the pairing is observable from one file. The right diagnostic question whenever the routing pattern changes: "if the LLM codepath reached this category, would `build_system_prompt` succeed?" If no, a short-circuit must protect it; if yes, the short-circuit is dead code and should be removed.
