# LLM Evaluation Pipeline: Implementation Guide

## Purpose and Audience

This guide is written for an LLM tasked with designing and implementing an evaluation pipeline for a movie metadata ingestion system. It is conceptual by design — it describes *what* to build and *why*, not *how* to wire it into a specific codebase. Before writing any code, produce a full implementation plan and data storage schema, and have that plan reviewed before proceeding.

---

## System Context

The system being evaluated is an ingestion pipeline that takes a movie as input and produces structured metadata outputs across multiple schema types. The pipeline can be run with different candidate configurations — varying models, system prompts, model parameters, and structured output schemas (all schemas share the same core parameters but may differ in details).

The evaluation pipeline must assess the quality of these outputs across all candidate configurations simultaneously, producing a per-dimension strength/weakness profile for each candidate rather than simply identifying a winner.

**Fixed parameters for this evaluation:**
- **Candidate pool:** Variable number of candidates (design for N ≥ 2 from the start)
- **Test corpus:** 70 movies, pre-selected for topic and data-sparsity diversity
- **Schema types:** 7 distinct generation types, each producing structured JSON output
- **Reference responses:** Generated by a stronger reference model per movie per schema — treated as a *reference*, not ground truth
- **Judge model:** A single Anthropic Claude model
- **Evaluation method:** Pointwise scoring (not pairwise)
- **Score scale:** 4-point per dimension
- **Runs per candidate/schema/movie:** Single run (no repeated sampling for consistency estimation)

---

## Scoring Paradigm: Pointwise, Not Pairwise

Use **pointwise scoring** as the primary evaluation method. In pointwise scoring, the judge evaluates each candidate output independently against a rubric and assigns a score per dimension. It does not compare two outputs side by side.

**Why not pairwise?**
- Pairwise comparisons are unstable: preferences flip at a much higher rate than absolute scores when distractor features are present.
- With N candidates, pairwise requires N×(N-1)/2 comparisons per sample, which compounds quickly.
- Pointwise directly produces the per-dimension strength/weakness profile required here.

**When to use pairwise as a supplement:** If two candidates end up within a narrow margin on aggregate scores, run targeted pairwise comparisons on the specific dimensions where they are closest. Use this only as a tiebreaker, not as the primary evaluation mechanism.

---

## The 4-Point Scale

Use a 4-point scale for all dimensions. Do not use 5-point or 10-point scales.

**Why 4 points?**
- Odd-numbered scales (1–3, 1–5) invite the judge to park in the middle, producing uninformative scores.
- Fine-grained scales (1–10) require discrimination the judge cannot reliably make.
- 4 points forces a lean: outputs must be above or below the midpoint, which produces more diagnostic data.

**Generic 4-point scale template** (to be instantiated with schema-specific criteria):

| Score | Label | Generic Definition |
|-------|-------|--------------------|
| 4 | Excellent | Fully meets the intent of this dimension. All expected content is present, accurate, and appropriately specified. No meaningful omissions or errors. |
| 3 | Adequate | Mostly meets the intent. Minor omissions, imprecision, or edge-case failures that do not materially affect usefulness. The output is deployable with low risk. |
| 2 | Partial | Meets some requirements but has meaningful gaps, inaccuracies, or structural problems that reduce usefulness. Requires correction before deployment. |
| 1 | Failing | Does not meet the intent of this dimension. Output is missing, degenerate, factually wrong, or structured in a way that is not usable. |

**When instantiating per schema:** Replace the generic definitions with concrete, observable criteria specific to that schema and dimension. The score label and structure stay the same; the definitions must be specific enough that two different judges would assign the same score to the same output. See the rubric design section for how to do this.

---

## Rubric Design: The Highest-Leverage Investment

**Do not write code before rubrics are finalized.** The quality of the evaluation pipeline is determined almost entirely by rubric quality. A well-implemented pipeline with weak rubrics produces noise. A well-designed rubric with a simple pipeline produces signal.

### Principles

**1. Schema-specific, not generic.**
Each of the 7 schema types requires its own rubric with its own scoring criteria. A rubric for `viewer_experience_metadata` (emotional palette, tone, disturbance profile) has fundamentally different quality criteria than one for `plot_events_metadata` (chronological accuracy, completeness, factual grounding). Generic rubrics like "rate quality 1–4" produce nearly useless data.

**2. Dimension-level granularity within each schema.**
Each schema should be broken into its constituent dimensions (individual fields or field groups), and each dimension should be scored separately. A model may produce excellent plot summaries but poor character relationship mappings. Aggregating these into a single schema-level score destroys diagnostic value.

**3. Concrete and verifiable criteria.**
Every score level must have a definition that is observable in the output — not a subjective impression. Avoid: "The output seems relevant." Prefer: "The output contains at least 3 narrative techniques that are identifiable in the film's source material, with no techniques that are generic or unverifiable."

**4. Distinguish objective from interpretive fields.**
Some fields have verifiable correct values (character names, plot events, release year). Others are genuinely interpretive (emotional palette, thematic elements, tone). Apply different scoring logic to each:
- **Objective fields:** Score on accuracy and completeness against verifiable facts. Errors are errors.
- **Interpretive fields:** Score on *defensibility* — does the output reflect a reasonable interpretation supported by the film's content? Do not penalize valid interpretive differences from the reference response.

**5. Encode verbosity expectations per schema.**
Some schemas benefit from more terms (e.g., `search_keywords`, `emotional_palette` — broader coverage is better). Others should penalize padding (e.g., `narrative_techniques` — generic filler is a quality defect). Explicitly state in each schema's rubric whether quantity contributes to quality, and if so, what the expected range is. This prevents the judge from applying its default verbosity preference inconsistently.

**6. Explicitly describe what a high-quality response looks like.**
For each dimension, the rubric must directly articulate what properties a good output has — not just what distinguishes a 4 from a 3. The judge's system prompt should contain affirmative descriptions: what should be present, what should be accurate, what range or depth is expected, and what distinguishes substantive content from generic filler. This anchors the judge's baseline without relying on examples. The reference response provided at evaluation time will serve as a calibration anchor for what quality looks like in practice for that specific movie and schema.

## Semantic Similarity

Many schema types in this pipeline are semantics-heavy — the outputs are not exact-match verifiable but should be semantically aligned with the reference response and the film's actual content.

**Semantic similarity must be a first-class scoring concern**, not an afterthought.

### How to incorporate semantic similarity

**In the rubric:** For interpretive and semantics-heavy dimensions, define score levels in terms of semantic coverage and alignment, not string matching. Example: a score of 4 means the output captures the same conceptual space as the reference response (even if expressed differently); a score of 2 means the output is partially on-topic but misses core semantic territory; a score of 1 means the output is semantically disconnected from what the film actually contains.

**In the judge prompt:** Explicitly instruct the judge to evaluate semantic content, not surface form. State that two outputs expressing the same meaning with different words should receive the same score. State that an output that uses the "right" words but misapplies them to a film's actual content should score lower than one that uses different but accurate words.

**For schemas where semantics dominate** (e.g., `emotional_palette`, `thematic_elements`, `viewer_experience_metadata`): Weight semantic alignment as the primary scoring criterion. Keyword overlap with the reference is a weak proxy; conceptual alignment with the film's actual content is what matters.

**For schemas where semantics are secondary** (e.g., `plot_events_metadata` with verifiable facts): Semantic similarity matters less than factual accuracy. Do not over-index on paraphrase matching when the field values are objectively verifiable.

## Judge Configuration

### Model selection
Use a single Anthropic Claude model as the judge throughout the evaluation. Do not switch judge models between runs — judge model changes invalidate comparisons.

**On self-enhancement bias:** The judge is Anthropic-family; all candidates are non-Anthropic. The bias therefore works against all candidates roughly equally rather than inflating one over another. Relative rankings between candidates are reliable. Absolute scores may be slightly depressed uniformly — account for this when interpreting results (compare candidates to each other, not to an abstract ideal). Rubric specificity is the primary mitigation: the more concrete the scoring criteria, the less room the judge has to apply stylistic preferences.

### Explicit chain-of-thought in output

**Require the judge to output its reasoning before its scores.** Structure the judge's output so that reasoning comes first, scores come second. This is not optional — it serves a diagnostic purpose that internal extended thinking cannot serve.

Rationale: You cannot inspect or debug internal extended thinking. If a score looks wrong, you need to know whether the rubric is flawed or the judge misapplied it. Explicit reasoning in the output gives you that audit trail. You will use it to refine rubrics between evaluation runs.

If using extended thinking (e.g., high reasoning effort settings), do so in addition to — not instead of — explicit output reasoning. They serve different purposes.

### Judge prompt structure

The judge prompt is split into a system prompt and a user prompt. These carry different content and serve different purposes.

**System prompt contains:**
1. **Role and task:** What the judge is doing and why
2. **Schema-specific rubric:** The full rubric for this schema type — dimension definitions, score-level criteria, verbosity expectations, and explicit description of what a high-quality response looks like for each dimension
3. **Scoring instructions:** Require dimension-by-dimension reasoning before dimension-by-dimension scores. Instruct the judge to evaluate semantic content, not surface form. Instruct the judge to treat the reference as a calibration anchor, not ground truth.
4. **Output format specification:** The structured output schema the judge must follow

**User prompt contains:**
1. **Original generation prompt:** The full user prompt that was sent to the candidate model, dynamically reconstructed using the same method that created it during ingestion. This gives the judge full context on what the candidate was actually asked to produce — no separate movie context field is needed since this is already embedded in the generation prompt.
2. **Reference response:** The Opus-generated reference output for this movie/schema, clearly labeled as a reference, not ground truth
3. **Candidate output:** The output being scored, clearly labeled

Do not include the reference response and candidate output in the same field — label them explicitly. The judge should be aware it is looking at a reference, not authoritative truth, especially for interpretive dimensions.

## Evaluation Flow

### Phase 0: Generate reference responses (run once, before any candidate evaluation)

```
For each movie in test corpus (70 movies):
  For each schema type (7 schemas):
    1. Reconstruct the generation user prompt for this movie/schema
    2. Run Claude Opus as the reference model → produce structured output
    3. Store reference response: (movie_id, schema_type, output, reference_model_id)
```

This phase must complete fully before any candidate evaluation begins. Reference responses are fixed for the duration of the evaluation. Do not regenerate them between candidate runs — doing so would introduce variability that undermines comparability.

### Phase 1: Generate and evaluate candidate outputs

```
For each candidate configuration (N candidates):
  For each movie in test corpus (70 movies):
    For each schema type (7 schemas):
      1. Reconstruct the generation user prompt for this movie/schema
      2. Run candidate model → produce structured output
      3. Store candidate output: (movie_id, schema_type, candidate_id, output, generation_metadata)
      4. Retrieve reference response for this movie/schema
      5. Call judge with:
           system prompt: schema rubric + scoring instructions
           user prompt: original generation prompt + reference response + candidate output
      6. Parse judge response → extract dimension-by-dimension reasoning and scores
      7. Store evaluation result: (movie_id, schema_type, candidate_id, dimension, score, reasoning, judge_metadata)
```

All candidate outputs and all evaluation results are stored in full. This allows manual spot-checking of both the raw generated outputs and the judge's scoring decisions independently.

### Reference responses
- Treat reference responses as a calibration anchor for interpretive dimensions, not as the correct answer. For objective dimensions (verifiable facts), the reference can be used more strictly, but still verify it is accurate.
- Do not expose the reference generation model's identity to the judge — label it only as "reference response."

### Candidate configuration management
Each candidate configuration must be fully specified before the run begins:
- Model identifier (provider + model name + version)
- System prompt (full text)
- Model parameters (temperature, top_p, max_tokens, etc.)
- Any provider-specific parameters
- Structured output schema in use

Store this configuration alongside results so every score can be traced back to the exact configuration that produced it.

## Data Storage: Plan Before You Run

**Design the data model and analysis queries before writing any pipeline code.** You will have up to `70 × N × 7 × K` score records (where K is the number of dimensions per schema). Without a planned schema and analysis structure, you will end up with a large flat table and no clear framework for interpreting it.

### Required data entities

**`candidates`** — one record per candidate configuration
- `candidate_id` (unique identifier)
- `model_id`, `provider`
- `system_prompt`, `parameters` (JSON)
- `schema_variant` (if schema differs between candidates)
- `run_timestamp`

**`reference_responses`** — one record per (movie, schema)
- `movie_id`, `schema_type`
- `output` (JSON, full structured output)
- `reference_model_id`

**`candidate_outputs`** — one record per (movie, schema, candidate)
- `movie_id`, `schema_type`, `candidate_id`
- `output` (JSON, full structured output)
- `generation_metadata` (latency, token counts, etc.)

**`evaluation_results`** — one record per (movie, schema, candidate, dimension)
- `movie_id`, `schema_type`, `candidate_id`, `dimension`
- `score` (1–4)
- `judge_reasoning` (text)
- `judge_model_id`
- `judge_call_metadata` (token counts, latency)

**`movies`** — one record per movie
- `movie_id`, `title`, `release_year`, `genre`
- `synopsis` (used in judge context)
- Any sparsity/diversity metadata used in selection

### Required aggregations (plan queries in advance)
- Per-candidate aggregate score by schema type
- Per-candidate aggregate score by dimension (across all schemas)
- Per-candidate aggregate score overall
- Per-movie variance across candidates (identifies movies where models diverge most)
- Per-schema variance across movies for a single candidate (identifies underspecified rubrics — high variance within a single model's scores on one schema is a rubric quality signal, not a model quality signal)
- Score distribution per dimension per candidate (to catch score clustering, which signals rubric issues)

## Common Pitfalls in This Scenario

**1. Rubric written at schema level, not dimension level.**
If the rubric scores the schema output as a whole rather than each dimension separately, you lose all diagnostic granularity. A model may score 3/4 overall because it excels on 4 dimensions and fails on 2. Schema-level scores hide this. Always score at the dimension level.

**2. Treating reference responses as ground truth for interpretive fields.**
If a film's emotional palette is described as "melancholic, bittersweet" in the reference and a candidate produces "somber, wistful" — that is not an error. Penalizing it as one introduces reference-anchoring bias. Distinguish objective from interpretive fields and score interpretive fields on defensibility, not match to reference.

**3. High per-schema score variance interpreted as model inconsistency.**
If one schema type shows unexpectedly high score variance across the 70 movies for all candidates, the most likely explanation is that the rubric for that schema is underspecified, not that all models are inconsistent on that schema. Variance within a single model's performance on one schema is a rubric quality signal. Diagnose it by reading the judge's reasoning for the high-variance movies.

**4. Implicit verbosity preference.**
Without explicit verbosity guidance in the rubric, the judge will apply its own preference — which may reward or penalize length inconsistently across schemas. Some schemas benefit from more terms; some do not. Specify this explicitly in every schema's rubric or the judge will decide on its own.

**5. No reasoning in judge output.**
Relying solely on internal extended thinking gives you a score with no audit trail. When a score looks wrong, you cannot determine whether the rubric is at fault or the judge misapplied it. Always require explicit reasoning in the structured output, before scores.

**6. Candidate configurations not fully pinned.**
If you cannot reproduce exactly what configuration produced a given set of scores, the results are not trustworthy for comparison purposes. Every candidate configuration — including system prompt, all parameters, and provider-specific settings — must be stored with the results.

**7. Score scale too coarse or too fine.**
A 3-point scale loses gradation. A 10-point scale produces unreliable discrimination. A 4-point scale forces the judge to take a position while remaining tractable. Do not adjust the scale after evaluation begins — doing so invalidates comparisons.

**9. Passing non-embeddable fields to the evaluator.**
Some schema outputs may include fields that are generated as intermediate reasoning or justification — content that explains or supports other fields but is not itself embedded in the vector index or reused by downstream metadata functions. Do not pass these fields to the judge. Only evaluate the fields that get directly embedded or consumed by other parts of the pipeline. Evaluating justification content conflates the quality of the model's reasoning process with the quality of its actual output, which are different things. Determine which fields are "live" output versus scaffolding, and strip scaffolding before the judge call.

**10. Running analysis after the fact with no structure.**
Having 70 × N × 7 × K score records with no planned aggregation queries means you will spend more time organizing data than interpreting it. The analysis structure must be designed before the pipeline runs.

## Pre-Run Checklist

Before running any evaluations:

- [ ] All 7 schema rubrics are written at dimension level with concrete score definitions and affirmative descriptions of what a high-quality response looks like
- [ ] Verbosity expectations are encoded per schema in the rubric
- [ ] Objective vs. interpretive fields are explicitly distinguished in each rubric
- [ ] Semantic similarity emphasis is correctly calibrated per schema
- [ ] For each schema, non-embeddable / justification fields are identified and will be stripped before judge calls
- [ ] Phase 0 is complete: reference responses generated by Claude Opus and stored for all 70 movies × 7 schemas
- [ ] All candidate configurations are fully specified and stored
- [ ] Judge system prompt template is finalized per schema; judge user prompt template reconstructs the original generation prompt dynamically
- [ ] Judge output requires explicit dimension-by-dimension reasoning before scores
- [ ] Data model is designed and all aggregation queries are written
- [ ] A small dry run (5–10 movies, 1–2 schemas) has been executed and both the raw candidate outputs and judge evaluations have been manually reviewed before committing to the full run
